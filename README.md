# ОТГ Audit

**Два Excel на вхід → один Excel на вихід, в якому:
  (1) усі імена й адреси зведені до єдиного формату,
  (2) 5 000 конкретних розбіжностей знайдено автоматично,
  (3) є юридично чистий аудит-слід (SHA-256, версії, список трансформацій).**

MVP-прототип для Innovate-хакатону: «Автоматизація обліку активів в ОТГ».
Працює на реальних даних Червоноградської ОТГ (42 тис. рядків). Повний
цикл — 5 секунд локально.

```
./run.sh           → http://localhost:8000
```

---

## Чого вимагає хакатон (PDF) і що з цим зробив продукт

| Вимога завдання | Реалізація |
|---|---|
| Імпорт Excel / CSV | openpyxl `read_only`, підтримка обох форматів |
| Нормалізація (ПІБ, адреси, ID) | 12 правил в `cleaner/normalize.py`, вся логіка описана в метаданих |
| **Автоматичний пошук розбіжностей** | **11 детекторів у `cleaner/detect.py`** — `critical`/`high`/`medium`/`low` |
| Виведення результату | П'ятий аркуш «Розбіжності» у вихідному xlsx + UI-банер з кількістю |
| UX, простота | Drop-two-files → Download. Жодних налаштувань |
| Демонстрація | 5 с на 42k рядків; файл відкривається в Excel/LibreOffice/Numbers |
| Правова чистота | On-premise, SHA-256 hash кожного вхідного файлу, аудит-лог |
| Метрика працездатності | UI показує: знайдено N розбіжностей, фін. ризик X грн |

## Архітектура

```
 ┌─────────────────┐
 │  Excel ДЗК      │──┐
 │  Excel ДРРП     │──┘      drop 2 files
 └─────────────────┘
         ↓
 ┌───────────────────────────────────────┐
 │ FastAPI  POST /api/clean              │
 └──────────┬────────────────────────────┘
            ↓
 ┌─────────────────┐   Stage 1  (cleaner/normalize.py)
 │ Normalizer      │   12 правил: ПІБ, адреси, ID, дати, частка,
 │                 │   `#VALUE!` errors, text format, latin→cyrillic
 └──────────┬──────┘
            ↓   (clean rows + change mask per cell)
 ┌─────────────────┐   Stage 2  (cleaner/detect.py)
 │ Detectors × 11  │   Працює на ВЖЕ очищених даних —
 │                 │   немає false positives від "вул." vs "вулиця"
 └──────────┬──────┘
            ↓   (findings з severity, evidence, row refs)
 ┌─────────────────┐   Stage 3  (cleaner/io_excel.py)
 │ xlsxwriter      │   5 аркушів: Власники, Земля, Нерухомість,
 │                 │   Розбіжності, Метадані
 └──────────┬──────┘
            ↓   (.xlsx bytes, ~4 MB)
 ┌─────────────────┐
 │  Download       │
 └─────────────────┘
```

**Ключове архітектурне рішення**: детектори запускаються **після**
нормалізатора, не паралельно. Наслідки:
- «Петренко» і «ПЕТРЕНКО» вже зведені, тож `name_mismatch` не дає false
  positives на косметичному шумі;
- адреси `"вулиця Дорошенка 31"` і `"вул. Дорошенка, буд. 31"` уже
  ідентичні, тож «два різних власники на одній адресі» — справжня
  розбіжність, а не різниця форматів;
- кожна розбіжність посилається на **рядок у тому ж xlsx**, тому
  бухгалтер одразу бачить оригінал після нормалізації.

## Що нормалізується (приклади з реальних даних)

### ПІБ

| Сире | Чисте |
|---|---|
| `Равлик Марiя Йосипiвна` (латинська i) | `Равлик Марія Йосипівна` |
| `ПЕТРЕНКО іван миколайович` | `Петренко Іван Миколайович` |
| `  Іваненко-Петренко  Іван ` | `Іваненко-Петренко Іван` |

### Адреси

| Сире | Чисте |
|---|---|
| `вулиця Дорошенка, будинок 31, квартира` | `вул. Дорошенка, буд. 31` |
| `Львівська область, м.Червоноград, вулиця Лемкiвська, 56` | `Львівська обл., м. Червоноград, вул. Лемківська, 56` |
| `вулиця Франка І., буд. 23, вулиця Франка І., буд. 23` (дубль) | `вул. Франка І., буд. 23` |
| `#VALUE!` | *(порожньо)* |

### Податкові номери

- Лише цифри; довжина поза межами 6–12 → відкидається.
- Колонка **явно помічена як Текст** у xlsx (Excel не сконвертує в число, нулі не відпадуть).
- Класифікація: 8 цифр → юр. особа, 10 цифр → фіз. особа, назва з «рада/держ/комунал» → орган влади (перекриває класифікацію за довжиною).

### Частка володіння

- Пусто / нерозпізнано → `1.0` + окрема булева колонка **«Частка невідома»** = `TRUE` для аудиту.

## Що детектимо (11 правил)

Кожен детектор — pure-функція в `backend/cleaner/detect.py`. Додати
правило — ~20 рядків.

| Рівень | Детектор | Що означає |
|---|---|---|
| critical | `missing_cadastral` | Запис без кадастрового номера — не можна ідентифікувати ділянку |
| critical | `missing_owner_id` | Немає ЄДРПОУ/РНОКПП — неможливо встановити платника |
| critical | `duplicate_cadastral` | Одна ділянка, два різних власники — подвійна реєстрація |
| high | `missing_area` | Немає площі — не можна нарахувати податок |
| high | `residential_no_building` | Житлова ділянка без жодного зареєстрованого об'єкта — незадекларована будівля |
| high | `name_mismatch` | Різне ПІБ при однаковому податковому номері |
| medium | `public_owner_as_private` | Орган влади зареєстровано як «Приватна» форма |
| medium | `share_overflow` | Сума часток > 100 % площі об'єкта |
| medium | `koatuu_address_mismatch` | Код КОАТУУ не відповідає області у адресі |
| low | `value_outlier` | Нормативна оцінка у 10× відхиляється від медіани |
| low | `ancient_reg_date` | Дата реєстрації до 1991 р. (ДРРП існує з 2013) |

## Аркуш «Метадані» — юридичний аудит-слід

Кожен вихідний файл містить окремий аркуш з:

- **Дата / час очищення** (UTC + локальний)
- **Версія скрипта** (семантично-версіонована, поточна 2.0.0)
- **SHA-256 обох вхідних файлів** — юридична прив'язка «цей звіт
  зроблено саме з цих байт»
- Розмір і назва файлів
- Статистика: рядків разом і окремо — змінених нормалізатором
- **Повний перелік з 13 застосованих трансформацій** дослівно
- Зведення розбіжностей по рівнях і сумарний фін. ризик

Для Держаудитслужби це критично: через два роки будь-хто може відкрити
файл і точно зрозуміти, **як і з чого** він був зроблений.

## Запуск

```bash
pip3 install -r requirements.txt
./run.sh
# → http://localhost:8000
```

Вимоги: Python 3.11+.

Залежності — **5 пакетів**:
- `fastapi` + `uvicorn` + `python-multipart` — web
- `openpyxl` — читання вхідних Excel
- `xlsxwriter` — швидкий запис (у ~25× швидше за openpyxl на нашому кейсі)

---

## Чому саме цей стек

- **FastAPI** — найшвидший шлях від прототипу до продакшна без спорів.
- **openpyxl read-only** — стабільно читає всі варіанти Excel, які
  вивантажує ДЗК/ДРРП.
- **xlsxwriter** — у 25× швидший за openpyxl на записі з per-cell
  стилями, файл на 39 % менший. Ми спочатку пробували openpyxl `write_only`
  (9,6 с), потім переключились на xlsxwriter (2,2 с на write).
- **Vanilla JS на фронті** — нема build-pipeline, нема npm-залежностей,
  бухгалтер на будь-якому Windows-ноутбуку може хостити це у себе.
