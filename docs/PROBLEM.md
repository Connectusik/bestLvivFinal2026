# Проблема, користувач і обмеження

## 1. Кого ми обрали як кінцевого користувача

**Начальник фінансового управління ОТГ (заступник голови з майнових
питань) — одна людина на громаду, загалом ~1 469 в Україні.**

Ми **свідомо не брали** в розгляд:
- «громаду в цілому» — не користувач, а стейкхолдер;
- мешканця-платника податку — на нього система не діє напряму;
- журналістів / антикорупційників — у майбутньому як public API, зараз скоуп відкидаємо;
- нотаріусів — інший правовий контур, інший продукт.

Чому саме ця людина:
- **має KPI у гривнях** — повнота місцевих майнових надходжень;
- **технічно не програміст** — потребує self-serve інструмент без бази даних;
- **приймає рішення про бюджет ІТ** — може сам купити підписку до 200 тис. грн без тендера (ЗУ «Про публічні закупівлі», ст. 3).

## 2. У якому процесі виникає проблема

Щорічна / квартальна інвентаризація комунального майна та звірка платників:

```
   1. Вивантажити Excel з ДЗК (земля)         ← раз на квартал
   2. Вивантажити Excel з ДРРП (нерухомість)  ← раз на квартал
   3. Відкрити обидва в Excel і VLOOKUP'ати за ЄДРПОУ/РНОКПП
   4. Шукати очима: «вулиця» vs «вул.», «i» vs «і», пропущені площі
   5. Формувати список підозрілих записів
   6. Передавати земле-/нерух-впоряднику для офлайн-перевірки
```

На 20 тис. рядків крок 3–5 займає 2–4 робочі дні на квартал. Жоден
VLOOKUP не знайде «Муличук» = «Мyличук» (латинська y) чи «вулиця
Дорошенка 31» = «вул. Дорошенка, буд. 31».

## 3. Чому існуючі інструменти не вирішують задачу

| Інструмент | Чому не підходить |
|---|---|
| **Prozorro** | Про закупівлі, не про майно. |
| **Дія** | Показує «мої» об'єкти, не агрегує по громаді. |
| **YouControl / Opendatabot** | Про юросіб; землю й нерухомість не звіряють. |
| **Excel + VLOOKUP** | Не нормалізує адреси, не масштабується. |
| **SAP / MEDoc** | Від 1 млн грн, не під специфіку ОТГ. |
| **Власний Python-скрипт бухгалтера** | Нема бухгалтера, який пише на Python. |

Порожня ніша: **крос-реєстр-аудит майнових активів ОТГ** з консистент­
ною нормалізацією. Заповнюємо саме її.

## 4. Обмеження

### Технічні
- ДЗК і ДРРП не мають публічного API — тільки ручний експорт у Excel.
- Формати колонок фіксовані, але **порядок і назви колонок однакові** з 2020 р.
- У вхідних даних — ~20 % «брудних» записів (див. приклади нижче).

### Юридичні
- Обробка ПІБ + адреси + РНОКПП = обробка персональних даних
  (ЗУ «Про захист персональних даних», ст. 2).
- Правова підстава обробки — **виконання обов'язку адміністрування
  місцевих податків** (ЗУ «Про місцеве самоврядування», ст. 26 + ПКУ).
- Згода платників не потрібна — дані вже публічні в держреєстрах.

### Організаційні
- Продукт має працювати локально (on-premise) для тих громад, які
  юридично не можуть пересилати дані в хмару.
- Команда продукту не має повноважень змінювати записи в ДРРП/ДЗК —
  тільки читати і пропонувати.

## 5. Як саме виглядає «брудний запис»

З реального датасету Червоноградської ОТГ (42 тис. рядків):

```
Сиру адресу:    "вулиця Франка І., будинок 23, вулиця Франка І., будинок 23"
Ми чистимо в:   "вул. Франка І., буд. 23"   (прибрали дубль)

Сире ПІБ:       "Равлик Марiя Йосипiвна"    (латинська i всередині)
Ми чистимо в:   "Равлик Марія Йосипівна"

Сиру адресу:    "#VALUE!"
Ми чистимо в:   ""                          (відкидаємо Excel-помилки)

Сира локація:   "Львівська область, м.Червоноград, вулиця Лемкiвська, 56"
Ми чистимо в:   "Львівська обл., м. Червоноград, вул. Лемківська, 56"
```

Після нормалізації детектори **бачать реальні**, а не артефактні розбіжності.
